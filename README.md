## ۱. تعریف و هدف پروژه

یه داشبورد تعاملی برای **پایش رسانه‌ای (Media Monitoring)** بیانیه‌های رسمی نظامی جمهوری اسلامی ایران دربارهٔ یه درگیری نظامی جاری، با دو منبع رسمی:
- **سپاه پاسداران انقلاب اسلامی** — عملیات موسوم به **«نصر ۲»**
- **ارتش جمهوری اسلامی ایران** — عملیات موسوم به **«صاعقه»**

هدف: نمایش مکان، تاریخ، نیروی عامل، و شرح هر حملهٔ ادعایی روی نقشه، با تأکید صریح بر این‌که **همهٔ داده‌ها ادعای یک‌طرفهٔ منابع رسمی است و راستی‌آزمایی مستقل ندارد**. این پروژه با «Rased»، زیرساخت پایش رسانه‌ای فارسی کاربر، مرتبطه.

## ۲. منابع دادهٔ اولیه (دستی، پیش از اتوماسیون)

مبنای تبدیل تقویم: تیر ۱، ۱۴۰۵ = ۲۲ ژوئن ۲۰۲۶.

## ۶. فاز ۲ — پایپ‌لاین خودکار همگام‌سازی (تکمیل و تثبیت‌شده)

### معماری نهایی
- **زمان‌بند**: GitHub Actions (`sync-sepahnews.yml`)، هر ۶ ساعت + دستی
- **مدل استخراج**: OpenRouter، مدل `openrouter/free` (روتر خودکار بین مدل‌های رایگان، پشتیبان از structured outputs)
- **منبع ۱ (سپاه)**: `sepahnews.ir/fa/rss/allnews` — فچ مستقیم (robots.txt مسدودکننده بود، با User-Agent شفاف جلو رفتیم)
- **منبع ۲ (ارتش)**: `aja.ir` — **مسدود در سطح شبکه** برای IPهای دیتاسنتر (نه فقط robots.txt) — هم GitHub Actions (Azure) و هم Cloudflare Worker با این مسدودیت برخورد کردن. راه‌حل نهایی: یه **VPS ایرانی** (`109.122.250.213:8787`) که یه پروکسی Node.js سبک (`systemd service`) اجرا می‌کنه؛ GitHub Actions از طریق این پروکسی (با هدر `X-Proxy-Secret`) به aja.ir وصل می‌شه. Secrets: `AJA_PROXY_URL`, `AJA_PROXY_SECRET`.

### جریان کار اسکریپت (`scripts/sync-rss.mjs`)، به‌صورت SOURCES-loop برای هر دو منبع
۱. فچ RSS (مستقیم یا از طریق پروکسی) → ۲. حذف تکراری (seen-guids.json، namespaced بر اساس منبع: `sepah:`/`artesh:`) → ۳. فیلتر ربط (کلیدواژهٔ عمومی + فعل عملیاتی) → ۴. فچ صفحهٔ کامل مقاله (نه فقط description) → ۵. استخراج بدنهٔ مقاله (پاکسازی نویز) → ۶. استخراج ساختاریافته توسط مدل → ۷. اعتبارسنجی چندلایه → ۸. `events.json` (موفق) یا `pending-review.json` + GitHub Issue (`needs-review`)

### استخراج بدنهٔ مقاله (متفاوت برای هر منبع)
- **سپاه**: بدون container مشخص در HTML؛ landmark متنی — شروع بعد از آخرین «دانلود همه تصاویر» (یا اولین «روابط عمومی سپاه پاسداران...» به‌عنوان fallback)، پایان در اولین «اشتراک گذاری» بعد از شروع (نویز هم قبل و هم بعد از بدنهٔ اصلی وجود داشت)
- **ارتش**: یه container واقعی و تمیز — `<span class="Content">` (نه `<td class="Content">` بیرونی که یه decoy بود) — با `node-html-parser` استخراج می‌شه، نه regex

### کد خبر
- سپاه: از regex روی `<link>` (`/news/(\d+)/`)
- ارتش: هیچ کد خبر انسانی‌خوانا در HTML وجود نداره (فقط GUID در نقش‌های فنی/plumbing) — همیشه خالی می‌مونه، طبق طراحی

## ۱۱. نکات مهم برای هر نشست جدید

- منبع واحد حقیقت: `/data/*.json` روی شاخهٔ `main`
- همهٔ تغییرات کد از طریق Claude Code اعمال می‌شه؛ این چت پرامپت می‌نویسه و نتیجه/لاگ رو بررسی می‌کنه
- قبل از merge، همیشه منتظر نتیجهٔ واقعی اجرای workflow بمونید، نه فقط ادعای موفقیت کد
- وقتی یه باگ جدید کشف می‌شه، اول **بررسی/دیباگ** (با اسکریپت‌های دیباگ موجود یا GitHub Actions) قبل از حدس زدن فیکس — این الگو بارها از دور اضافی فیکس جلوگیری کرده
- هر داده‌ای که وارد داشبورد می‌شه، ادعای یک‌طرفهٔ منبع نظامیه و باید همیشه به همین شکل نمایش داده بشه
- دو منبع (سپاه/ارتش) معماری متفاوتی برای استخراج بدنهٔ مقاله دارن (landmark vs container) — این تفاوت رو نباید فراموش کرد اگه منبع سومی اضافه بشه
